## **Objective:**

The primary objective of this Refined QATD is to document test execution evidence, root cause analysis, and verified fixes for all defects identified during the Preliminary Round (SEC-001, WF2-001, WF3-001), and to validate the three major architectural enhancements introduced since that round: **(1) Auto-categorization** — receipts are now automatically mapped to active reimbursement policies using `PolicyReimbursableCategory` lookups, eliminating manual category selection; **(2) AI-powered auto-approval and auto-rejection** — the Compliance Agent can now autonomously decide claim outcomes based on confidence scores and budget checks, triggering automatic payouts or rejection notifications; **(3) Xendit payment integration** — approved claims (whether by HR or AI) automatically initiate disbursements to employee bank accounts via Xendit's test-mode API. This document demonstrates that Reclaim's three-workflow AI pipeline operates reliably under both nominal and degraded-LLM conditions, that all RBAC and workflow-blocking defects have been resolved, and that the new autonomous decision-making and payment systems function correctly before the live judging demo.

## **5. Live Demo / UAT Risks**

_Operational boundaries judges should know before testing the application._

- **Risk 1 — LLM Rate Limits (GLM + Gemini Fallback):** The system automatically falls back from GLM-5.1 to `google/gemini-3.1-flash-lite-preview` (via OpenRouter) if GLM is unavailable. However, OpenRouter has its own per-minute rate limits. **Do not run more than 3 concurrent claim submissions during the demo.** If the TopNav notification banner displays _"AI model degraded — using Gemini backup model,"_ this is expected behaviour and not a crash — wait 5 minutes for the TTL to clear, then proceed normally.

- **Risk 2 — Receipt Image Quality & File Size:** OCR accuracy degrades with low-resolution, blurry, or heavily compressed receipts. The `visual_anomalies_detected` flag will fire and route the claim to Bucket B (Requires Attention) for HR review. **Use clean, well-lit JPEG receipts under 2 MB for all demo uploads** to demonstrate the happy-path OCR flow. Larger files are accepted but will increase `ProcessingScreen.tsx` wait time.

- **Risk 3 — Policy Upload Wait Time:** WF1 (Policy Studio) involves a 5-node LangGraph pipeline: PDF conversion → category extraction → condition extraction → DB save → pgvector embedding. In production mode this takes **4–7 seconds** and is represented by a three-step animated overlay (Uploading → Analyzing → Saving). **Click "Upload Policy" only once and wait for the overlay to dismiss** before navigating away. Clicking twice will submit the pipeline twice.

- **Risk 4 — Browser Compatibility:** This prototype was developed and QA-tested exclusively in **Google Chrome (desktop, 1440px)**. Safari and Firefox may experience minor layout differences on the HR triage dashboard filter panel and the claims verification dual-panel. Mobile browsers are supported for the Employee portal (PWA-enabled, iOS tested) but the HR Portal is optimised for desktop only.

- **Risk 5 — Xendit Mock Mode Webhook Timing:** In mock mode, payout webhooks are simulated with a fixed 10-second delay. During the demo, **wait at least 15 seconds after approving a claim** before checking for the `PAID` status. The mock webhook simulator fires on a background task and may take additional time depending on server load. Real Xendit webhooks in production would arrive asynchronously within 1–5 seconds.

- **Risk 6 — Employee Bank Details Required for Payout:** Auto-approval will trigger automatic payouts **only if the employee has configured bank details** in Settings (bank code + account number). If testing the full payout flow, ensure demo employees have banking details pre-seeded via `backend/add_employee_bank_details.py` or entered manually. Otherwise, approved claims will remain in `APPROVED` status without payout initiation (this is expected behaviour, not an error).

- **Risk 7 — Auto-Approval Confidence Threshold:** The AI auto-approves claims when `confidence ≥ 0.8` AND the claim is within budget. To demonstrate the auto-approval flow, use clear, low-risk receipts (e.g., standard F&B receipts under RM 50 within a RM 500 monthly budget). High-risk or ambiguous claims will route to HR for manual review (this is intended safety behaviour).

- **Risk 8 — Multiple Policy Groups:** Submitting receipts from multiple categories (e.g., "F&B" + "Transport") will create **multiple separate reimbursements**, one per policy group. The frontend will display multiple claim results after analysis. This is correct behaviour — do not interpret it as a system error. Each reimbursement will be evaluated and approved/rejected independently.
